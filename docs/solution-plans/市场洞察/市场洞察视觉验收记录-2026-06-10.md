## 2. 总体结论

Insight 当前页面已具备试点可用性：导航完整、核心数据能展示、桌面端主要页面没有明显遮挡或崩坏，移动端有底部导航兜底。

但体验上还不能算“非常完善”。首轮视觉验收发现若干影响用户信任和效率的问题，其中 P0/P1 建议先修再扩大试点范围。

## 3. 问题清单

| 编号 | 优先级 | 页面 | 现象 | 影响 | 建议 |
| --- | --- | --- | --- | --- | --- |
| VQA-1 | P0 | 全局头部 | 管理员登录后管理后台显示 Administrator，但 Insight 头部仍显示“张伟”和“张”头像。源码定位：`frontend/src/app/insight/layout/InsightHeader.tsx` 直接写死。 | 身份展示错误，影响用户信任，也会影响企业微信跳转后的用户感知。 | 改为读取 `useAuthStore` 当前用户，缺失时显示真实兜底；头像取姓名首字或工号，不写死。 |
| VQA-2 | P0 | 首页、数据源、报告中心 | 页面加载期间直接显示 `0`、`共 0 份`、`正在读取数据源...`，随后才变成真实值；移动端数据源页初始显示 0，等待后变为 27。 | 用户会误判系统无数据或权限异常。 | 统计卡和列表在 query loading 时显示骨架屏或“读取中”，不要把 `undefined` 当 0 展示。 |
| VQA-3 | P1 | 首页看板 | KPI 主数和副数口径容易矛盾，例如“数据源 27 / 已启用 0”。 | 管理者第一眼会困惑，不知道是数据源总数、可见数还是启用数。 | 复核 dashboard API 指标口径；副指标改成明确文本，如“启用 27 / 停用 0”或隐藏无意义副数。 |
| VQA-4 | P1 | 移动端报告中心 | 首屏几乎全是表单，主按钮“生成报告”和报告历史需要滚动才能看到。 | 报告页缺少主动作引导，移动端用户不知道下一步在哪里。 | 移动端保留核心字段，折叠高级选项；生成按钮做底部 sticky 操作条。 |
| VQA-5 | P1 | 移动端情报中心 | 筛选器占满首屏，情报列表露出很少。 | 高频浏览/审核效率低。 | 移动端默认只展示关键词和状态 Tab，高级筛选折叠。 |
| VQA-6 | P1 | 情报中心 | 候选审核操作在桌面端主要表现为小图标，视觉上不易判断“通过/驳回/忽略”。 | 审核属于高风险操作，误点成本高。 | 桌面端改为文字按钮或图标+文字；驳回/忽略增加确认或撤销提示。 |
| VQA-7 | P1 | 质量运营 | 失败原因直接展示原始技术异常，如 `'NoneType' object is not subscriptable`、长 URL 和 HTTP 错误。 | 业务用户难以理解，也拉低页面专业感。 | 后端或前端做错误归类：超时、解析失败、外部服务失败、素材不足；原始错误放详情抽屉。 |
| VQA-8 | P1 | 移动端质量运营 | 加载态大卡片占据整屏；加载完成后第一屏仍只能看到采集质量前几项，失败排行和跳转入口很靠后。 | 排障效率弱。 | 加载态改紧凑骨架；移动端质量页增加顶部摘要和“查看失败原因”锚点。 |
| VQA-9 | P1 | 移动端数据源配置 | 第一屏被操作按钮和统计卡占满，数据源列表需要继续滚动；表格在移动端不适合阅读。 | 数据源配置在移动端可用但不高效。 | 移动端切卡片列表，隐藏次要列，把“测试/日志/更多”收进操作菜单。 |
| VQA-10 | P2 | 登录页 | 页脚仍显示 `© 2025 AI 智能平台`。 | 时间过期，影响细节可信度。 | 改为当前年份或 `2025-2026`。 |

## 4. 建议修复顺序

1. VQA-1：修复 Insight 头部用户身份硬编码。
2. VQA-2、VQA-3：修复加载态误导和 KPI 口径。
3. VQA-4、VQA-5：优化移动端报告/情报首屏。
4. VQA-6：优化候选审核操作表达和误点防护。
5. VQA-7、VQA-8：质量运营错误信息业务化和移动端排障体验。
6. VQA-9：数据源移动端卡片化。
7. VQA-10：登录页年份细节。
