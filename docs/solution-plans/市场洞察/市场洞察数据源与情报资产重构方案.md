# 市场洞察数据源与情报资产重构方案

## 一、调整目标

当前数据源配置把 800 多个底层执行源直接暴露给用户，维护成本高，也不符合业务人员的使用习惯。后续需要改成分层设计：底层先沉淀所有可用来源，上层再按企业、主题、报告和 AI 助手使用这些来源。

本轮重构不要求一次性完成所有爬取能力。第一阶段重点是先把来源管理清楚，形成可维护的渠道库和执行源基础，再逐步建设企业监测、AI 自动评审、RAG、知识图谱和战略分析能力。

## 二、总体分层

建议按五层建设：

1. 渠道库
   维护具体网站、数据库、平台和资料来源，例如 FoodDaily、食品伙伴网、WIPO、CNIPA、东方财富、同花顺、上交所、深交所、企业官网等。

2. 执行源
   系统内部用于真实搜索、抓取、解析和调度的技术对象。执行源可以由渠道库、企业档案和主题配置自动生成，默认不直接暴露给普通用户维护。

3. 监测配置
   面向用户，包括企业监测、行业主题、政策主题、技术专利主题、电商新品主题、自定义主题等。

4. AI 自动评审
   取消人工审核概念，由 AI 按企业类型、主题规则和业务提示词判断资料进入正式情报、候选情报或噪声归档。

5. 情报资产层
   正式情报、候选情报、原文证据、结构化标签、向量索引、关键词索引、图谱关系统一沉淀，供报告生成、AI 助手、深度研究和后续战略分析调用。

## 三、第一阶段先做渠道库

第一阶段先把来源放上来，不急着保证每个渠道都能完整爬取。渠道库属于系统基础配置，放在 Insight 系统设置中，仅管理员可维护，普通业务用户不直接看到该配置。

渠道库用于回答：

- 这个来源是什么？
- 属于哪类渠道？
- 适合服务哪些监测场景？
- 当前支持哪种采集方式？
- 是否需要登录、授权、接口或人工导入？
- 当前状态是可采集、待接入、暂不支持还是仅作搜索限定？

### 渠道库建议字段

| 字段 | 说明 |
| --- | --- |
| 渠道名称 | 如 FoodDaily、食品伙伴网、CNIPA、东方财富 |
| 渠道类型 | 企业官网、行业媒体、财经资讯、政策监管、专利技术、综合新闻、电商平台、公众号、资料库 |
| 渠道地址 | 官网、搜索页、栏目页或数据库入口 |
| 适用场景 | 企业新闻、经营财经、行业资讯、政策文件、专利技术、电商新品、自定义主题 |
| 采集方式 | 搜索发现、列表页抓取、详情页抓取、API、RSS、文件导入、人工导入、待接入 |
| 是否需要登录 | 不需要、需要账号、需要授权、未知 |
| 当前接入状态 | 已支持、部分支持、待接入、暂不支持 |
| 默认可信度 | 高、中、低 |
| 默认频率 | 每日、每周、手动 |
| 默认处理策略 | AI 自动评审、仅候选、暂不入库 |
| 备注 | 反爬、验证码、付费墙、版权限制、接入建议 |

### 渠道分组建议

1. 企业官网
   企业官网、新闻中心、投资者关系、产品中心、技术文章等。

2. 行业媒体
   FoodDaily、食品伙伴网、食品饮料产业研究、粮油市场报、每日食品、食品产业观察市场信息网、食业头条等。

3. 财经资讯与公告
   东方财富、同花顺、雪球、财联社、中国证券网、中证网、证券日报、证券时报、上交所、深交所、第一财经、每日经济新闻等。

4. 政策监管
   市场监管总局、食品药品监管、政府网站、标准公告、人民网、新华网、央视新闻等。

5. 专利技术
   WIPO、CNIPA、专利数据库、技术资料库、论文或标准类来源。

6. 综合资讯
   新浪、网易、搜狐、腾讯、澎湃、36 氪、虎嗅、光明网等。

7. 电商与新品
   电商搜索、品牌旗舰店、公开商品页、食品新品报道来源。

## 四、第二阶段整理执行源

执行源是系统内部对象，不应作为用户主维护入口。执行源配置同样归入 Insight 系统设置，仅管理员可维护；普通业务用户后续只面对企业监测、主题监测、情报中心和报告中心。

现有 800 多个数据源可以保留，但需要归到更上层的监测配置和渠道库下。

执行源建议增加或补齐这些归属字段：

| 字段 | 说明 |
| --- | --- |
| 渠道 ID | 来自哪个渠道 |
| 监测配置 ID | 属于哪个企业监测或主题监测 |
| 监测对象 ID | 企业、行业主题、政策主题等 |
| 执行角色 | 企业新闻、官网动态、经营财经、专利技术、电商新品、政策监管等 |
| 生成方式 | 系统生成、用户创建、历史迁移 |
| 采集策略 | 轻量、标准、深度、结构化 |
| 当前状态 | 启用、停用、异常、待接入 |

用户默认看到的是监测配置；只有管理员在系统设置中排障、高级配置或查看执行日志时，才进入执行源明细。

## 五、第三阶段设计企业监测

企业监测面向业务用户，核心不是数据源，而是企业关系和 AI 判断规则。

### 企业类型

企业档案需要维护企业关系类型：

- 客户
- 竞对
- 潜在客户
- 供应商
- 合作伙伴
- 其他

不同企业类型默认关注点不同。

### 默认关注点

客户重点关注：

- 新品、产能、采购变化、渠道变化、配方升级、成本压力、经营风险、扩产、合作、招标、质量问题、政策影响。

竞对重点关注：

- 新品发布、价格策略、渠道动作、营销活动、专利技术、产能扩张、客户合作、融资并购、战略转向、风险事件。

潜在客户重点关注：

- 业务增长、产品线变化、新建项目、融资扩张、供应链需求、采购线索、招聘动向、渠道扩张、合作机会。

供应商重点关注：

- 价格波动、产能变化、质量风险、环保安全、交付风险、政策影响、财务经营、替代供应可能性。

### 企业监测操作逻辑

新增企业监测时，用户只需要完成：

1. 选择或新增企业档案。
2. 选择企业类型。
3. 确认企业识别名称，包括全称、简称、英文名、品牌名、常见缩写。
4. 选择监测模块，例如企业新闻、官网动态、经营财经、专利技术、电商新品、政策监管、自定义关键词。
5. 选择监测强度：轻量、标准、深度。
6. 选择频率：每日、每周、手动。
7. 确认 AI 情报规则，可使用默认提示词，也可单独调整。
8. 保存后由系统生成底层执行源。

### 企业级 AI 提示词

每个企业可以有自己的 AI 评审提示词，默认继承企业类型模板。

客户类默认提示词：

```text
你是研发营销市场洞察平台的企业情报评审助手。当前企业是我司客户或潜在合作客户。请重点保留与我司研发、销售、客户经营和产品机会相关的信息，包括新品上市、配方变化、采购需求、产能扩张、渠道布局、经营变化、质量风险、政策影响、合作动态和供应链变化。与企业无关、重复、泛泛新闻、股票短讯、招聘噪声、无业务价值内容，不进入正式情报。
```

竞对类默认提示词：

```text
当前企业是我司竞对。请重点保留其新品发布、技术专利、价格策略、渠道动作、营销活动、产能扩张、客户合作、融资并购、战略方向、风险事件等信息。需要判断该信息对我司产品、市场、研发、销售或战略是否有参考价值。无关转载、低价值财经快讯、重复内容、泛泛介绍不进入正式情报。
```

潜在客户类默认提示词：

```text
当前企业是我司潜在客户。请重点识别可能形成销售线索、合作机会或研发配套机会的信息，包括业务扩张、新产品、新工厂、新渠道、融资、采购需求、配方升级、供应链调整、招标、招聘和市场动作。没有明确业务机会或关联价值的信息不进入正式情报。
```

供应商类默认提示词：

```text
当前企业是我司供应商或潜在供应商。请重点识别价格波动、产能变化、交付风险、质量风险、环保安全、政策影响、财务经营、替代供应可能性等信息。与供应稳定性、成本、质量、合规无关的信息不进入正式情报。
```

## 六、第四阶段取消人工审核，改为 AI 自动评审

后续不再突出人工审核流程，采集结果统一进入 AI 自动评审。

AI 评审输出三类结果：

1. 正式情报
   符合企业或主题规则，有明确业务价值，置信度达到阈值。

2. 候选情报
   可能有价值，但证据不足、相关性一般、置信度不足或需要后续补抓正文。

3. 噪声归档
   明显无关、重复、验证码页、纯股价短讯、百科泛信息、广告页、低价值转载。

AI 评审建议输出结构：

```json
{
  "decision": "formal",
  "score": 0.86,
  "reason": "涉及客户新品扩张，可能带来配料需求",
  "intelligence_type_code": "product_update",
  "intelligence_type": "产品动态",
  "business_value": "销售机会",
  "tag_codes": ["sales_opportunity", "new_product", "functional_sugar"],
  "tag_names": ["销售机会", "新品扩张", "功能糖"],
  "suggested_new_tags": ["儿童营养新品"],
  "related_products": ["功能糖", "植物蛋白"],
  "opportunities": ["可跟进低糖配料方案"],
  "risks": [],
  "entities": ["企业A", "产品B"],
  "evidence": "原文关键摘录"
}
```

标签和分类采用受控字典：情报类型从平台内置口径选择，业务标签从 `insight_tag` 启用标签中选择。AI 不得自由生成正式标签；模型发现字典缺口时只写入 `suggested_new_tags`，由管理员维护后再进入正式标签体系。

## 七、第五阶段建设情报资产层

采集来的资料必须能被报告、AI 助手、深度研究、企业画像、战略分析复用。

每条资料至少沉淀三部分：

1. 原始证据
   URL、标题、正文、发布时间、来源渠道、抓取时间、原文摘要、正文片段。

2. 结构化情报
   企业、企业类型、产品、技术、政策、渠道、事件类型、机会点、风险点、业务价值、置信度、相关部门。

3. 检索索引
   向量索引、关键词索引、标签索引、时间索引、企业索引、权限索引。

## 八、RAG 与知识图谱方向

### RAG

RAG 建议分三层索引：

- 原文片段向量索引
- 正式情报摘要向量索引
- 结构化字段过滤索引

报告生成和 AI 助手检索时，应先按权限、企业、企业类型、时间、情报类型、产品标签过滤，再做向量召回。

默认报告优先使用正式情报；候选情报只有用户明确选择“包含候选线索”时参与。

### 知识图谱

第一阶段图谱不做复杂推理，先沉淀轻量关系。

节点：

- 企业
- 产品
- 技术
- 政策
- 行业
- 渠道
- 事件
- 关键词
- 内部客户或销售区域

关系：

- 企业 发布 产品
- 企业 申请 专利
- 企业 涉及 政策
- 企业 合作 企业
- 企业 扩产 项目
- 产品 属于 品类
- 情报 证明 事件
- 内部客户 关联 外部企业

后续可用于回答：

- 哪些竞对正在布局低糖方向？
- 哪些客户最近可能有配料升级需求？
- 某项政策影响哪些客户和产品线？
- 外部市场动作和内部销售变化是否存在关联？

## 九、与内部经营数据结合

后续可以把内部经营数据接入情报资产层，包括：

- 客户销售额
- 产品销量
- 订单变化
- 毛利变化
- 区域销售
- 客户拜访记录
- CRM 商机
- 投诉或质量反馈

组合分析示例：

```text
外部情报：客户新品增长、扩产、渠道扩张。
内部数据：该客户近 3 个月采购下降。
AI 分析：可能存在供应替代风险，建议销售跟进。
```

```text
外部情报：竞对推出低糖新品。
内部数据：我司相关产品销售增速放缓。
AI 分析：可能受竞品影响，需要关注价格、渠道和配方卖点。
```

## 十、建议实施顺序

### 阶段 1：渠道库

- 新增渠道库模型和管理页面，入口放在 Insight 系统设置，仅管理员可见。
- 先导入现有网址清单。
- 标记渠道类型、采集方式、接入状态和适用场景。
- 不要求每个渠道马上能爬取。

### 阶段 2：执行源归组

- 现有数据源继续保留。
- 增加渠道、监测配置、监测对象、执行角色等归属字段。
- 将历史 800 多个数据源迁移到对应企业或主题下。
- 执行源配置入口放入 Insight 系统设置，仅管理员可见；普通用户前端默认隐藏底层执行源。

### 阶段 3：企业监测

- 企业档案增加企业关系类型。
- 企业监测支持按企业启用模块。
- 支持企业级 AI 评审提示词。
- 支持批量设置企业类型、模块、频率和提示词。

### 阶段 4：AI 自动评审

- 用 AI 评审替代人工审核主流程。
- 输出正式情报、候选情报和噪声归档。
- 将企业类型提示词、企业级提示词、主题提示词纳入评审上下文。

### 阶段 5：情报资产化

- 正式情报和候选情报进入统一资产层。
- 增加向量索引、关键词索引和结构化过滤索引。
- 让报告生成、AI 助手、深度研究统一从资产层取证据。

### 阶段 6：知识图谱和内外部数据分析

- 从企业、产品、技术、政策、事件中抽取轻量图谱。
- 接入内部经营数据。
- 支持客户机会、竞对风险、政策影响和战略分析类问题。

## 十一、当前需要先确认的设计点

1. 渠道库是否作为第一步先做。
2. 是否同意把现有数据源降级为执行明细。
3. 企业关系类型是否先采用客户、竞对、潜在客户、供应商、合作伙伴、其他。
4. AI 评审是否采用正式情报、候选情报、噪声归档三类。
5. 报告生成和 AI 助手是否优先只读取正式情报，候选情报作为可选范围。
