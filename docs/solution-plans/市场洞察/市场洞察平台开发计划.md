# 研发营销市场洞察平台开发计划

INSIGHT DEVELOPMENT PLAN

开发顺序建议围绕一条主线推进：先做通用联网采集，让系统能从百度、通用网页、Bocha/博查 API 和本地 Firecrawl 获取真实情报；再做清洗入库、权限可见和情报池；最后完成报告模板、偏好、自定义图表和运营增强。这样第一版就不是静态 Demo，而是具备真实信息获取能力的情报系统。

阶段策略**P0 / P1 / P2**

优先目标**联网采集闭环**

开发对象**前端 + 后端 + 数据库**

更新时间**2026-06-06**

## 1. 开发策略

### 先通用采集，后特殊渠道

P0 先做百度搜索、通用网页抓取、Bocha/博查多源查询和本地 Firecrawl 正文抽取；公众号、财报站点、登录态站点等特殊渠道后续逐步补。

### 先可控联网，后规模化采集

第一版支持关键词任务、手动触发、限量抓取、去重和审核，不直接做大规模无人值守爬取。

### 先结构化，后智能化

企业、主题、标签、权限、素材关系先结构化，AI 输出作为增强层，不让模型决定数据边界。

建议每个模块都按“采集通道 → 清洗入库 → 后端接口 → 前端接入 → 验证”的节奏推进。采集层先求稳定、可追溯、可重试，不追求覆盖所有特殊网站。

### P0-10 数据源配置可用化细化

这一阶段的目标不是简单保存几个 URL，而是把“采集来源 + 发现策略 + 筛选策略 + 执行周期”组合成可复用的数据源配置。后续新增公众号、财报站、交易所公告、行业网站、登录态站点时，只新增适配器和配置 Schema，不重写采集主流程。

| 能力 | 设计要点 | 第一版范围 |
| --- | --- | --- |
| 数据源类型 | 用 `source_type` 区分通道，用 `fetch_config` 保存通道参数，避免为每个来源硬编码字段。 | `official_site` 官网、`web_page` 通用网页、`baidu_news` 百度资讯、`bocha_news` 博查资讯、`bocha_web` 博查网页搜索。 |
| 百度资讯 | 百度查询默认应走资讯结果，而不是普通网页结果；适配器需要显式生成资讯搜索 URL，例如包含 `tn=news` 或等价参数。 | 关键词输入后只解析资讯结果列表，保留标题、来源、发布时间、摘要、原始 URL，再进入正文抓取。 |
| 关键词配置 | 一个数据源可维护多个独立搜索关键词，执行时逐个关键词分别查询搜索引擎，避免把企业名和多个意图词拼成一个分散注意力的复合 query。 | 先做关键词数组、排除词、每词最大发现数、每词抓取上限、近 24 小时/近 7 天/不限时间；企业监控建议填写企业名、品牌名、简称和英文名。 |
| 定时任务 | 数据源保存采集周期、下次执行时间、上次执行结果；调度器只读取启用数据源并创建采集任务。 | 支持手动、每 15 分钟、每小时、每天、自定义 cron；第一版可先保存配置并提供“立即执行”，再接后台调度。 |
| 后处理规则 | 搜索阶段以覆盖资讯为优先，只做 URL 去重、排除词和上限保护；抓取正文后先做质量规则标记，再由 LLM 生成摘要、分类、标签、置信度和价值判断。 | 每个数据源可配置摘要提示词、最低价值分数、是否自动入候选池；低价值资讯不在搜索阶段丢弃，而是在候选或后处理状态中标记；正文过短、安全验证页、疑似股价快讯、标题高度相似等会形成质量标签或自动忽略。 |
| 导入与批量维护 | 支持从 Excel/CSV 导入数据源和关键词，便于一次性维护企业官网、竞品关键词、行业关键词。 | 导入字段包括名称、类型、URL、关键词、排除词、频率、状态、主题类型、主体名称、筛选提示词。 |
| 执行日志 | 每次执行要能看到发现多少、过滤多少、抓取多少、入库多少、失败多少，以及失败原因。 | 数据源详情页展示最近任务、命中 URL、过滤原因、候选情报链接，并可跳转到情报中心只查看该数据源产生的候选情报。 |
| 扩展边界 | 新增特殊渠道时，只需要实现搜索/发现适配器、正文抽取适配器和配置表单片段。 | 预留公众号、财报 PDF、交易所公告、政策站点、登录态站点、RSS、站内搜索。 |

P0-10 建议拆成三步：第一步做数据源 CRUD、关键词和立即测试；第二步做百度资讯/博查资讯通道和导入；第三步做周期配置、执行日志和 LLM 筛选。这样既能快速可用，又不会把调度、AI 和采集适配器耦合死。

2026-06-04 第一版已推进：新增真实数据源 CRUD、启停、软删除、列表筛选、批量文本导入和立即执行接口；前端数据源配置页已接入真实 API，可维护官网、通用网页、百度资讯、博查资讯、博查网页搜索和多源资讯数据源。百度资讯通道已显式走资讯搜索参数，数据源可保存关键词组、包含词、排除词、抓取数量、抓取前 N 条、采集周期、Cron 和 LLM 筛选提示词。2026-06-05 补齐数据源最近执行日志第一版：新增 `GET /api/v1/insight/data-sources/execution-logs`，复用 `insight_task` 展示搜索发现、网页抓取任务的状态、进度、发现数、抓取数、候选数和失败原因；前端数据源配置页右侧已接入“最近执行日志”。同日继续补齐筛选审计链路：搜索发现任务会把过滤前数量、规则保留数、去重保留数、LLM 保留数、最终保留数、保留结果、过滤原因和通道错误写入 `insight_task.output_payload`；前端“最近执行日志”支持展开查看筛选明细。后续按真实采集策略调整：数据源执行会逐个关键词独立搜索，搜索阶段不再默认使用 LLM 拦截结果，而是收集资讯链接、URL 去重并抓取正文；LLM 摘要、分类、标签和价值评分后置到候选资讯生成阶段。情报中心改为资讯流卡片，一条 URL 对应一条候选或正式情报，并提供原文跳转链接。2026-06-05 继续补充采集质量治理和测试闭环：正文抓取后记录质量评分、低质量原因、疑似重复候选，AI 标签与质量规则标签一并进入候选；数据源配置页可跳转到情报中心按数据源筛选候选审核结果。P0 封板补充：数据源表单开放关联企业 ID，采集候选自动带 `company_id`；正式情报列表和详情页可加入报告素材池，供 P1 报告草稿读取；后端提供遗留任务清理接口和 P0 验收脚本，封板前需要确认无超时 running/pending 任务。

### P1-1 企业档案最小闭环进度

## 8. 风险与取舍

### 先做通用爬虫，不先做特殊爬虫

百度、Bocha/博查和 Firecrawl 能覆盖第一版联网查询；公众号、财报站点、登录态站点等作为后续适配器逐步加。

### 权限必须先做

情报数据有选择性可见，这是底层能力，后做会影响所有接口和页面。

### 报告不要一步到位

先做可保存的报告草稿，再做模板、偏好、图表、导出，节奏会稳很多。

2026-06-06 P1 进入封板验收：新增只读验收脚本 `backend/scripts/insight_p1_acceptance.py`，用于检查企业档案、数据源、候选情报、正式情报、报告素材池、报告、报告引用和遗留任务状态，不会重新生成报告或写入测试数据。验收通过后，P1 的报告主链路可视为第一版闭环；P1-5 后续只保留“手工录入、编辑和排序图表配置”的增强任务。
