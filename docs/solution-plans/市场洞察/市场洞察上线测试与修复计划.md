## 1. 使用规则

本文件作为 Insight 上线前的主控清单。后续每次修复、测试、验收都需要同步更新本文件。

记录规则：

1. 每开始一个阶段，先在对应阶段的“执行记录”追加开始时间、目标和负责人。
2. 每完成一个修复项，更新该项状态，并记录改动文件、验证命令和结果。
3. 每发现阻塞或风险，写入“风险与待决策事项”。
4. 每次阶段验收后，写入“阶段验收结论”，明确是否可进入下一阶段。
5. 不允许只在对话里确认完成，必须把修复过程沉淀到本文档。

状态标记：

| 状态 | 含义 |
| --- | --- |
| 未开始 | 尚未进入修复或测试。 |
| 进行中 | 正在修复或测试。 |
| 已修复 | 代码或配置已完成，等待或已完成验证。 |
| 已验证 | 已通过本阶段要求的测试。 |
| 阻塞 | 存在外部依赖、方案分歧或环境问题。 |

## 3. 上线修复与测试顺序

上线修复顺序固定为：

1. 权限与数据隔离。
2. 核心业务闭环稳定。
3. 调度与失败恢复。
4. 报告交付能力。
5. 通知推送。
6. 系统设置与运营配置。
7. 质量运营看板。
8. 上线总回归。

原则：先修安全边界，再修业务主链路，再修运营体验；不要在权限未闭环前扩展新采集渠道。
