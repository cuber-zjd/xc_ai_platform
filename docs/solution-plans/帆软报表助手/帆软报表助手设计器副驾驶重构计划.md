# 帆软报表助手设计器副驾驶重构计划

## 1. 背景与结论

帆软报表助手后续不应先做一套抽象的“项目管理平台”，而应从真实报表文件出发：先对接 MinIO，读取现有 FineReport 报表文件；再解析 `.cpt` / `.frm` 等类 XML 结构化文件；然后在前端实现类帆软设计器渲染；最后围绕 SQL、样式、布局、填报、筛选、超链接等功能逐项 AI 化。

结论：采用“报表文件优先”的落地路线。产品术语统一使用“报表”，不再用“项目”作为用户侧主概念。第一阶段目标不是让 AI 生成完整报表，而是让平台能看到、读取、解析和渲染已有报表文件。

## 2. 核心原则

- 用户感知对象是“报表”，不是“项目”。
- AI 不直接生成或修改 CPT/XML；AI 只生成 SQL、ReportDSL / 中间结构补丁、操作建议和解释。
- CPT/XML 的读取、解析、写回和生成必须由确定性程序完成。
- 先实现真实文件读取和渲染，再做 AI 改稿能力。
- 每次读取或写回报表文件都记录 hash、路径、修改时间和来源，避免覆盖 FineReport 设计器手工修改。
- 平台内 AI 修改和 FineReport 设计器内手工修改都必须纳入同一个报表快照体系，不能互相静默覆盖。
- 轻量渲染结果只是平台预览，不等同于 FineReport 运行时预览。
- 生成或写回文件默认仍只能进入 `webroot/APP/reportlets/AI生成报表/` 专用预览目录，正式 reportlets 发布后续单独设计审批流程。

## 3. 保留资产

- `backend/app/services/agent/fr_report/fr_minio_service.py`：帆软专用 MinIO 访问入口，使用 `FR_AI_MINIO_*`，与平台通用 `MINIO_*` 隔离。
- `backend/app/services/agent/fr_report/minio_staging_service.py`：复用帆软专用 MinIO 访问和 staging 上传思路，但需要扩展为报表文件读取、列表、hash 检测能力。
- `backend/app/services/agent/fr_report/cpt_generator.py`：继续作为确定性 CPT 生成 / 写回能力基础。
- `backend/app/services/agent/fr_report/preview_validator.py`：继续负责 FineReport 预览校验。
- `backend/app/services/agent/fr_report/sqlserver_query_service.py`：作为 SQL 校验和样例数据预览基础。
- `backend/app/schemas/agent/fr_report/report_dsl.py`：保留为平台中间结构的参考，但后续需要向 `FrReportDocument` 演进。
- 旧三步生成接口短期保留，后续作为历史生成入口或迁移入口。

## 4. 术语调整

- `报表`：用户侧核心对象，对应一个或多个 MinIO 报表文件。
- `报表文件`：MinIO 中的 `.cpt`、`.frm` 或相关资源文件。
- `报表快照`：某次读取、解析、AI 修改、设计器同步或写回后的版本记录。
- `报表文档结构`：平台解析 CPT/XML 后得到的中间结构，建议命名为 `FrReportDocument`。
- `操作补丁`：AI 或人工产生的结构化修改，例如改 SQL、改列宽、合并单元格、增加筛选项。

## 5. 目标交互形态

页面仍采用“类设计器工作台 + AI 副驾驶”：

- 左侧资源区：MinIO 报表目录、报表文件列表、数据集、字段树、快照历史。
- 中间画布区：根据真实 CPT/XML 解析结果渲染类帆软网格。
- 右侧属性区：单元格属性、数据绑定、样式、筛选、填报、超链接。
- AI 副驾驶区：通过聊天触发明确功能，不直接改文件。

第一版的关键体验是：用户能打开一个 MinIO 中已有报表，并在平台里看到尽量接近帆软设计器的结构化渲染。

## 6. 数据模型草案

### 6.1 `fr_report`

报表主表。记录报表名称、业务分类、当前文件路径、当前快照、来源目录、状态、所有者和权限范围。

建议字段：

- `report_id`
- `report_name`
- `report_type`
- `current_file_path`
- `current_snapshot_id`
- `owner_user_id`
- `status`
- `metadata_json`

### 6.2 `fr_report_file`

报表文件表。记录 MinIO 中发现或导入的文件。

建议字段：

- `file_id`
- `report_id`
- `bucket`
- `object_path`
- `file_name`
- `file_ext`
- `file_size`
- `file_hash`
- `last_modified`
- `source`
- `is_staging`

### 6.3 `fr_report_snapshot`

报表快照表。记录某一次读取、解析、AI 修改、设计器同步或写回后的状态。

建议字段：

- `snapshot_id`
- `report_id`
- `file_id`
- `parent_snapshot_id`
- `snapshot_no`
- `source`：`minio_scan` / `designer` / `ai` / `imported` / `rollback`
- `file_hash`
- `document_json`
- `parse_status`
- `compatibility_status`
- `change_summary`
- `preview_url`

### 6.4 `fr_report_operation`

结构化操作表。记录 AI 或人工产生的操作补丁。

操作类型示例：

- `update_sql`
- `update_dataset_binding`
- `update_cell_text`
- `merge_cells`
- `set_cell_style`
- `set_column_width`
- `set_row_height`
- `set_filter_param`
- `set_form_property`
- `set_hyperlink`
- `write_back_cpt`
- `sync_external_change`

### 6.5 `fr_report_dataset`

报表数据集表。记录从 CPT/XML 解析出的数据集、SQL、字段、参数和校验结果。

### 6.6 `fr_report_external_sync`

外部设计器同步表。记录 FineReport 设计器或其他外部工具对 CPT 的修改检测结果，包括检测路径、检测 hash、基础快照、同步状态、解析状态和处理结果。

## 7. 中间结构 `FrReportDocument`

后端解析 CPT/XML 后统一输出 `FrReportDocument`，前端只面向该结构渲染，不直接理解 CPT/XML。

第一版建议支持：

- `reportMeta`：标题、文件名、路径、更新时间、备注。
- `sheets`：工作表列表。
- `grid`：行列数量、默认行高、默认列宽。
- `cells`：单元格文本、公式、字段绑定、样式引用。
- `merges`：合并单元格范围。
- `styles`：字体、颜色、边框、对齐、背景。
- `datasets`：数据集名称、SQL、字段、参数。
- `parameters`：筛选参数和默认值。
- `forms`：填报控件和基础属性。
- `hyperlinks`：超链接目标和触发单元格。
- `warnings`：解析不支持项和兼容性提示。

不要求第一版完整覆盖 FineReport 所有能力，但必须保留原始节点引用或路径，便于后续渐进解析和写回。

## 8. 接口草案

### 8.1 MinIO 报表文件

- 当前已落地入口：`GET /api/v1/fr/ai-reports/files`，沿用现有帆软助手路由，列出 MinIO 允许目录下的 `.cpt` / `.frm` 报表文件。
- 当前已落地结构读取入口：`GET /api/v1/fr/ai-reports/files/structure`，在线内存读取 MinIO CPT，不落盘；第一版返回 XML 根节点、FineReport 版本、数据集、连接名、参数、SQL 片段和解析 warnings。
- 规划资源域入口：`GET /api/v1/fr/reports/files`，后续报表资源模型稳定后可迁移或并存。
- `POST /api/v1/fr/reports/files/scan`：扫描报表目录并刷新文件索引。
- `GET /api/v1/fr/reports/files/{file_id}`：读取文件元信息。
- `GET /api/v1/fr/reports/files/{file_id}/download`：下载原始报表文件。

### 8.2 报表解析与渲染

- `POST /api/v1/fr/reports/files/{file_id}/parse`：解析 CPT/XML，生成报表快照。
- `GET /api/v1/fr/reports/{report_id}`：读取报表详情。
- `GET /api/v1/fr/reports/{report_id}/snapshots`：读取快照列表。
- `GET /api/v1/fr/reports/{report_id}/snapshots/{snapshot_id}`：读取快照和 `FrReportDocument`。

### 8.3 外部修改同步

- `GET /api/v1/fr/reports/{report_id}/sync/status`：检查当前 MinIO 文件 hash 是否与最新快照一致。
- `POST /api/v1/fr/reports/{report_id}/sync/import`：将 FineReport 设计器手工修改同步为新快照。
- `POST /api/v1/fr/reports/{report_id}/sync/ignore`：忽略一次外部变更。

### 8.4 AI 能力入口

- `POST /api/v1/fr/reports/{report_id}/ai/sql`：SQL 生成、修改、解释。
- `POST /api/v1/fr/reports/{report_id}/ai/style`：样式和布局修改。
- `POST /api/v1/fr/reports/{report_id}/ai/filter`：筛选区修改。
- `POST /api/v1/fr/reports/{report_id}/ai/form`：填报属性修改。
- `POST /api/v1/fr/reports/{report_id}/ai/hyperlink`：超链接配置修改。
- `POST /api/v1/fr/reports/{report_id}/operations/apply`：应用结构化操作并生成新快照。
- `POST /api/v1/fr/reports/{report_id}/write-back`：确定性写回 CPT 到 staging。

## 9. 分阶段实施计划

### 阶段一：MinIO 报表文件读取

- 梳理当前 FineReport 报表文件所在 MinIO bucket 和目录。
- 保持帆软 MinIO 与平台通用 MinIO 配置隔离：帆软报表使用 `FR_AI_MINIO_*`，合同、知识库、图标等平台文件继续使用 `MINIO_*`。
- 后端新增报表文件读取服务，列出 `.cpt` / `.frm` 等文件。
- 记录文件大小、路径、修改时间和 ETag；需要更强一致性时再下载文件计算 hash。
- 前端将当前 Demo 左侧资源区改为真实报表文件列表。
- 前端提供用户级显示范围配置，可选择文件夹或报表；后端只保存用户选择显示的路径，空列表表示显示全部，避免存储全量报表清单。

验收标准：页面能展示 MinIO 中真实报表文件，点击文件能看到文件元信息、ETag 和修改时间；用户能保存自己的显示范围，并可随时恢复全部显示。

### 阶段二：CPT/XML 解析器

- 后端读取 MinIO 报表文件内容。
- 建立 `FrReportDocument` 基础 Schema。
- 先解析 sheet、行列、单元格文本、合并单元格、基础样式、公式、数据集引用。
- 对暂不支持的节点记录 warnings，不丢原始信息。

验收标准：选择一个真实 CPT 后，后端能返回结构化 `FrReportDocument`，并标明解析覆盖范围和不支持项。

当前进展：已完成阶段二第一小步，后端可在线读取 UTF-8/XML CPT 并解析数据集、连接名、参数、SQL 片段和版本信息；前端右侧副驾驶区已展示结构摘要和数据集 SQL。后续继续补单元格、合并、样式、控件和超链接解析。

### 阶段三：类帆软渲染器

- 前端根据 `FrReportDocument` 渲染网格。
- 支持行列头、单元格、合并单元格、基础样式、公式和字段占位。
- 支持单元格选中，并在右侧属性区展示解析出的属性。
- 画布在 2K、普通桌面和较窄桌面下自适应填满内容区。

验收标准：真实报表文件能在平台里渲染出可辨认的类帆软设计器效果，且页面无明显溢出、遮挡和大面积留白。

### 阶段四：报表快照和外部修改检测

- 每次解析生成 `fr_report_snapshot`。
- 记录文件 hash 和来源。
- 打开报表时检查 MinIO 文件 hash 是否变化。
- 检测到 FineReport 设计器外部修改时，提示同步为新快照。

验收标准：用户在 FineReport 设计器中修改报表文件后，平台能检测到 hash 变化，并允许同步为新快照；平台不会静默覆盖外部修改。

### 阶段五：SQL AI 化

- 从 `FrReportDocument.datasets` 中识别数据集和 SQL。
- AI 支持 SQL 生成、修改、解释和字段映射建议。
- SQL 结果先更新中间结构和快照，不直接写 CPT。
- SQL 校验继续走受控数据源和样例数据预览。

验收标准：用户能通过 AI 修改一个报表数据集 SQL，并看到 SQL diff、校验结果和新快照。

### 阶段六：样式和布局 AI 化

- AI 支持列宽、行高、合并、字体、颜色、边框、对齐、区域位置等修改。
- AI 输出结构化操作补丁。
- 后端校验补丁并应用到 `FrReportDocument`。
- 前端展示操作差异，用户确认后生成新快照。

验收标准：用户能通过 AI 修改报表样式和布局，画布即时反映结果，并保留可回滚快照。

### 阶段七：筛选、填报、超链接 AI 化

- 解析和展示筛选参数。
- 解析和展示填报控件、校验规则和提交目标。
- 解析和展示超链接。
- AI 支持新增、修改和解释这些配置。

验收标准：用户能通过 AI 调整筛选区、填报属性和超链接配置，且修改结果可在属性区和快照中追踪。

### 阶段八：确定性写回 CPT 和 FineReport 预览

- 将修改后的 `FrReportDocument` 确定性写回 CPT。
- 写回前检查当前 MinIO 文件 hash，发现外部变更时阻止默认覆盖。
- 写回 staging 后返回 FineReport 预览地址。
- 展示写回 warnings、预览状态和产物路径。

验收标准：平台修改后的报表能写回 staging，并可打开 FineReport 预览；遇到外部修改时不会默认覆盖。

### 阶段九：旧三步流程迁移和收口

- 旧三步任务迁移为报表和快照。
- 旧页面降级为历史生成入口或跳转到新报表工作台。
- 清理重复逻辑，保留兼容接口一段时间。
- 完善文档、验收脚本和回归测试。

验收标准：新入口承担主流程，旧任务可查看或迁移，核心链路稳定。

## 10. 风险与控制

- CPT/XML 结构复杂：第一版只解析核心可渲染能力，对不支持项明确 warnings。
- 渲染与 FineReport 不一致：平台渲染定位为“类设计器结构预览”，FineReport 运行时仍以官方预览为准。
- 外部修改覆盖风险：写回前必须比较 hash，发现外部设计器修改时默认同步为新快照，不自动覆盖。
- CPT 反向解析不完整风险：保留原始节点引用，不承诺完整还原所有 FineReport 能力。
- AI 输出不可控：AI 只输出结构化操作补丁，由后端校验并确定性应用。
- SQL 安全风险：只允许受控数据源、只读查询、字段和行数限制、超时控制。
- 旧任务兼容风险：先并行新能力，不直接删除旧表和旧接口。

## 12. 验证方式

- 后端：`cd backend && uv run pytest`。
- 后端静态检查：`cd backend && uv run ruff check .`。
- 前端构建：`cd frontend && pnpm build`。
- 前端专项检查：对帆软报表助手页面运行 ESLint 和浏览器截图验证。
- MinIO 验收：扫描真实 bucket 和目录，确认能列出报表文件和 hash。
- 解析验收：选择真实 CPT，确认返回 `FrReportDocument` 和 warnings。
- 渲染验收：前端渲染真实报表结构，检查行列、合并、文本、样式和滚动体验。
- 外部同步验收：手工修改 MinIO 报表文件后刷新平台，确认 hash 变化被检测并可同步为新快照。
- AI 验收：按 SQL、样式、布局、筛选、填报、超链接逐项验证。
