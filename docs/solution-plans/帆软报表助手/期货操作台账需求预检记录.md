# 期货操作台账需求预检记录

## 背景

期货操作台账报表由 AI 报表助手根据用户上传的 Excel 模板和自然语言需求生成。当前业务口径要求先完成需求预检，再进入 SQL、ReportDSL 和 CPT 生成流程。

## 已确认口径

- 基础资料维护表按一行一条合约配置维护。
- `账户名称 + 合约品种 + 合约代码` 作为基础资料维护表组合主键。
- `吨数/手`、操作单位、状态等属性挂在上述组合主键下。
- 录入时开仓和平仓在同一行维护，不拆分独立平仓表。
- 操作台账通过 `账户名称 + 合约品种 + 合约代码` 关联基础资料。
- 查询截止日收盘价独立维护，用于浮动盈亏计算。

## 建议表结构

| 表名 | 用途 | 关键字段 |
| --- | --- | --- |
| `fr_future_contract_base` | 期货合约基础资料 | `account_name`、`contract_variety`、`contract_code`、`tons_per_lot` |
| `fr_future_trade_ledger` | 期货操作台账 | `ledger_id`、`account_name`、`contract_variety`、`contract_code`、开仓字段、平仓字段 |
| `fr_future_settlement_price` | 每日收盘价维护 | `price_date`、`contract_code`、`settlement_price` |

## 生成约束

- 新建报表未显式选择模板时，默认使用 `webroot/APP/reportlets/数据分析/农产品价格平台/大豆/谷物大豆国际海运费报价-周报.cpt` 作为样式和参数栏参考模板。
- 需求预检返回的维护表必须体现组合主键。
- SQL 生成不得引入独立平仓表。
- SQL JOIN 必须包含 `t.account_name = b.account_name`、`t.contract_variety = b.contract_variety`、`t.contract_code = b.contract_code`。
- 查询 SQL 必须绑定 `start_date`、`end_date`、`account_name`、`contract_variety`、`contract_code` 参数；参数为空时应走默认范围或不过滤，保证 FineReport 首次打开可预览。
- CPT 需要生成顶部参数栏，日期参数使用 `DateEditor`，维度参数使用 `ComboBox` 并通过 `TableDataDictionary` 从主数据集取值，查询按钮使用 `FormSubmitButton`。
- 期货台账表格必须保留两层表头：基础字段纵向合并，`开仓`、`平仓`、`持仓` 横向合并；字段顺序固定为账户、合约、策略、开仓、平仓、持仓、操作单位。
- 同一行开平仓模式下，平仓数量不得超过开仓数量。
- AI 生成的数据模型属于候选结构，正式接入前仍需用户确认真实数据库表名、字段名和权限。
- 如果使用 AI 设计的候选结构进行 FineReport 预览，必须先在 FineReport 数据连接对应数据库中创建候选表；否则预览会报“对象名无效”。

## 填报改造记录

- ReportDSL 新增 `writeBack` 配置，用于描述填报控件、写回表、写回字段、主键字段和行操作按钮。
- 期货台账启用填报时，CPT 生成器生成 `ReportWebAttr`、`ReportWriteAttr`、单元格控件、顶部提交工具栏和行内插入/删除按钮。
- `ledger_id` 作为隐藏主键列，列宽为 0；提交时使用 `=if(len({ledger_id}) = 0, uuid(), {ledger_id})` 为新增行生成主键。
- 计算字段不写回，包括平仓盈亏、持仓方向、剩余持仓、收盘价和浮动盈亏；这些字段继续由 SQL 查询计算。
- 填报报表的 FineReport 预览地址必须使用 `op=write`，否则会按普通查看模式打开，无法显示提交、校验、添加记录和删除记录工具栏。
- 期货台账填报单元格不使用主数据集下拉字典，避免填报预览时每行控件反复绑定 `ds_main` 造成页面卡顿；首版只使用文本、数字和日期控件。

## 平台化闭环

- 需求预检结果新增结构化 `writeBackPlan`，页面可以直接展示是否启用填报、写回表、主键、隐藏键、可编辑字段、计算字段、增删行权限和安全策略。
- 第二步 ReportDSL 生成会消费 `writeBackPlan`，自动生成 `writeBack`、控件、行操作和隐藏主键，不再依赖人工手工改代码。
- 通用填报需求会先进入 `needs_confirmation` 状态，要求确认写回表、主键和可编辑字段；未确认主键时不得自动发布到正式目录。
- 通用填报默认采用轻量控件策略：文本、数字、日期；单元格不默认绑定主数据集下拉，避免性能问题。
- 期货操作台账作为已沉淀场景，预检可直接给出生产候选方案：隐藏 `ledger_id` 主键、写回 `fr_future_trade_ledger`、支持插入行和删除行。

## 期权操作台账补充口径

- 场内期权操作台账作为独立场景识别为 `option_operation_ledger`，不能再落入期货模板。
- 期权模板的多层表头需要解析为“开仓日期、开仓权利金单价、平仓日期、平仓权利金单价”等完整语义字段，空白尾列不得进入字段列表。
- 建议维护 `fr_option_contract_base` 和 `fr_option_trade_ledger` 两张候选表；基础表维护品种合约、标的合约、期权类型、执行价、买/卖和合约乘数，台账表同一行记录开仓和平仓。
- `fr_option_trade_ledger.ledger_id` 作为隐藏主键；可编辑字段包含项目字段、开仓字段、平仓字段、持仓量和备注；`realized_profit` 为只读计算字段。
- 期权收益情况默认公式为 `(平仓权利金单价 - 开仓权利金单价) * 开仓成交量 * 合约乘数 - 开仓手续费 - 平仓手续费`，合约乘数未维护时按 10 的业务假设处理。
- 期权台账 DSL 需要保留两层表头：`项目`、`开仓`、`平仓`、`持仓量`、`备注`，并沿用默认样式模板与轻量填报控件策略。
- 期权候选表结构必须使用英文下划线命名，并统一包含 `id` 主键；用户可在第一步选择 `id` 是否使用数据库自增。
- 第一阶段 DDL 生成支持 `sqlserver`、`mysql`、`postgresql` 三种方言：SQL Server 使用 `sys.sp_addextendedproperty`，MySQL 使用表/字段 `COMMENT`，PostgreSQL 使用 `COMMENT ON TABLE/COLUMN`。
- 用户可通过 `source_table_name` 快捷指定主表名，也可通过 `table_name_overrides_json` 指定 `fr_option_trade_ledger` 和 `fr_option_contract_base` 的真实表名；无效表名应回退到默认英文下划线表名。

## 验证方式

- 调用需求预检接口，检查基础资料表 `keys` 为 `account_name`、`contract_variety`、`contract_code`。
- 调用 SQL 生成接口，检查推荐表只包含三张候选表且不包含独立平仓表。
- 检查生成 SQL 使用三字段关联基础资料，并保留平仓数量约束。
- 使用期权 Excel 模板调用需求预检接口，检查场景为 `option_operation_ledger`，推荐表为 `fr_option_contract_base`、`fr_option_trade_ledger`，字段中包含 `开仓权利金单价` 和 `平仓权利金单价`。
- 在 `XcTest` 数据连接库中创建三张候选表并插入样例数据后，调用第三步生成 CPT，确认 FineReport 预览页面不再出现 `11300001` 或“对象名无效”。
- 填报改造后，检查 CPT 包含 `ReportWriteAttr`、`AppendRowButton`、`DeleteRowButton`、`TextEditor`、`NumberEditor`、`DateEditor` 和 `PreviewType="1"`，且报表主体区域不包含单元格级 `ComboBox`。
- 使用 `op=write` 预览地址打开报表，确认页面出现提交、数据校验、添加记录和删除记录工具栏。
